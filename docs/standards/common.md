# Common Standards

## 目的

この規約は、家計簿アプリのコード品質、設計方針、実装スタイルを統一するための共通ルールである。

フロントエンド、バックエンド、DB、テストコードのすべてで、実装者や生成AIによって書き方や設計がぶれないようにすることを目的とする。

特に以下を重視する。

- ファイルごとの書き方のずれを防ぐ
- 仕様書と実装のずれを防ぐ
- 責務の混在を防ぐ
- 将来の変更に強いコードにする
- CodexなどのAIエージェントが迷わず実装できる状態にする

---

## 最優先ルール

以下はすべての実装で必ず守る。

- 仕様書にない機能を勝手に追加しない
- 単一責務の原則を守る
- コメント対象は領域別規約を優先する。フロントエンドはTSDoc、バックエンドはGoDocを必須とする
- コメントは「コードが何をしているか」ではなく「何のために存在するか」「どういう意図で使うか」を説明する
- 認証済みユーザーの `user_id` をリクエストボディやクエリから受け取らない
- 他ユーザーのデータを取得・更新・削除できる実装にしない
- 秘密情報、アクセストークン、パスワードをログに出さない
- lint / formatter / test が通る状態を保つ

---

## コメント方針

### 基本方針

コメント対象は、各領域の詳細規約に従う。

- フロントエンドは `docs/standards/front-end/index.md` に従い、関数、Reactコンポーネント、custom hook、usecase、mapper、API関数、type、interfaceへTSDocを書く
- バックエンドは `docs/standards/back-end/go.md` に従い、関数、method、type、interfaceへGoDocを書く
- DBは `docs/standards/db/migrations.md` に従い、migration目的コメントとDB COMMENTを残す

この共通規約の「必要に応じて」は、領域別規約が存在しない対象にだけ適用する。

コメントはコードの逐語的な説明ではなく、以下が分かる内容にする。

- 何のために存在するのか
- どの画面や処理で使うのか
- どのような意図や制約があるのか
- なぜその責務として分離しているのか

### 悪いコメント例

以下のように、コードを読めば分かる内容をそのまま書くコメントは禁止する。

```ts
// amountをnumberに変換する
const amount = Number(input);
```

```go
// Expenseを作成する
func CreateExpense(...) {}
```

### 良いコメント例

処理の目的や利用意図が分かるコメントを書く。

```ts
// 出費入力欄の文字列を、APIに送信できる整数の金額へ正規化する
export const normalizeExpenseAmount = (input: string): number => {
  // ...
};
```

```tsx
// トップ画面で最短導線の出費登録を行うための金額入力コンポーネント
export const ExpenseAmountInput = () => {
  // ...
};
```

```go
// ログインユーザーの出費を登録し、トップ画面更新用の月次サマリーを返す
func CreateExpense(...) {}
```

### コメントと実装をずらさない

コメントは実装意図を固定するために書く。

以下のようなコメントは不要とする。

- 変数名や関数名だけで意図が明確なもの
- 処理内容を1行ずつ説明するもの
- 実装と乖離しやすい詳細すぎる説明

コメントと実装が矛盾する場合は、必ずコメントも更新する。
コメント規約を機械チェックできる場合は、対象領域の `check` コマンドに含める。

## 単一責務の原則

### 基本方針

関数、class、Reactコンポーネント、hook、usecase、handler、repositoryは、単一責務の原則を必ず守る。

1つの単位に複数の責務を詰め込まない。

責務が増えた場合は、適切な単位に分割する。

### 禁止する実装

以下は禁止する。

- UI表示、API通信、データ整形を1つのコンポーネントにまとめる
- handler内にDB操作や集計ロジックを書く
- repository内に業務ロジックを書く
- usecase内にHTTP request / responseの詳細を書く
- 1つの関数で入力検証、DB保存、レスポンス整形、ログ出力をすべて行う
- 1つのファイルに無関係な関数や型を詰め込む
- 汎用コンポーネントに特定画面の業務ロジックを混ぜる

### 分割の目安

以下の責務は、必要に応じて分ける。

- UI表示
- 入力状態管理
- 入力値の正規化
- バリデーション
- API通信
- 業務処理
- DB操作
- レスポンス変換
- エラー変換
- ログ出力

### 判断基準

以下に当てはまる場合は、責務が大きすぎる可能性が高い。

- 関数やコンポーネント名が `And` を含みたくなる
- 1つの関数が複数の外部依存を直接扱っている
- テストしたい観点が複数に分かれる
- 変更理由が複数ある
- 画面仕様の変更とAPI仕様の変更の両方で同じコードが変わる

## 設計方針

### 仕様書優先

実装は以下の仕様書を正とする。

- `docs/README.md`
- `docs/product/spec.md`
- `docs/product/*.md`
- `docs/features/*.md`
- `docs/architecture/*.md`
- `docs/api/*.md`
- `docs/db/*.md`
- `docs/front-end/spec.md`
- `docs/back-end/spec.md`
- `docs/db/spec.md`
- `docs/standards/agent-workflow.md`
- `docs/standards/common.md`
- `docs/standards/front-end/index.md`
- `docs/standards/back-end/index.md`
- `docs/standards/db/index.md`

仕様に迷った場合は、実装で勝手に決めず、仕様書を更新してから実装する。

### 依存方向を守る

各レイヤーの依存方向を守る。

フロントエンドでは、以下の責務分離を守る。

```txt
app
  ↓
features
  ↓
domains / libs
```

バックエンドでは、オニオンアーキテクチャの依存方向を守る。

```txt
interface
  ↓
usecase
  ↓
domain

infrastructure は domain repository interface を実装する
```

### DTOとdomain型を混ぜない

API request / response DTO と、アプリ内で使うdomain型を無闇に混ぜない。

必要に応じて、usecaseや変換関数でDTOからdomain型へ変換する。

### 業務ロジックの正はバックエンドに置く

残額計算、固定費の対象月判定、日別集計、年間集計などの正となる業務ロジックはバックエンドに置く。

フロントエンドでは、表示や入力補助のための整形に留める。

## 命名規則

### 共通

名前は役割が分かるように具体的にする。

曖昧な名前を避ける。

避ける名前の例：

```txt
data
item
value
result
temp
handleData
process
```

ただし、短いスコープのループ変数など、意味が明確な場合は例外とする。

### ファイル名

ファイル名は各技術スタックの慣習に従う。

- TypeScript / ReactのコンポーネントはPascalCase
- TypeScriptの通常ファイルはcamelCase
- Goのファイル名はsnake_case
- Markdownファイルはkebab-caseまたは既存構成に合わせる

### 関数名

関数名は「何をするか」が分かる名前にする。

良い例：

```txt
normalizeExpenseAmount
createExpenseUsecase
getMonthlySummary
validateIncomeInput
```

悪い例：

```txt
handle
process
doSomething
convert
```

### Boolean名

Booleanは状態が分かる名前にする。

良い例：

```txt
isActive
isLoading
isAuthenticated
includedInBalance
```

悪い例：

```txt
flag
check
status
```

## エラーハンドリング方針

### 共通

エラーは握りつぶさない。

ユーザーに表示するエラーと、ログに残すエラーを分けて考える。

### ユーザー向けメッセージ

ユーザー向けには、分かりやすい日本語メッセージを表示する。

内部エラーの詳細やスタックトレースを画面に出さない。

### ログ

ログには調査に必要な情報を残す。

ただし、以下はログに出さない。

- access token
- password
- secret key
- service role key
- 個人情報を過剰に含むデータ

---

## セキュリティ方針

### 認証・認可

未ログイン状態ではアプリ画面を利用できない。

APIでは必ずログインユーザー本人のデータのみ取得・更新・削除する。

`user_id` はリクエストボディやクエリパラメータから受け取らない。

### 秘密情報

秘密情報をフロントエンドに置かない。

`.env` に置く値は、フロントエンドで公開してよいものとバックエンドだけで扱うものを明確に分ける。

### 入力値

フロントエンドのバリデーションを信用しない。

バックエンドでも必ず入力値を検証する。

---

## Lint / Format / Test 方針

### 基本方針

実装後は、対象範囲のlint、format、testを実行する。

少なくとも以下が通る状態で完了とする。

- フロントエンドのlint
- フロントエンドのformat
- フロントエンドのtest
- バックエンドのformat
- バックエンドのlint
- バックエンドのtest

### エラーを放置しない

lintやtestの失敗を残したまま完了しない。

どうしても解消できない場合は、原因、未解決内容、次に確認すべきことを明記する。

### 自動修正の扱い

formatterやlintの自動修正を利用してよい。

ただし、自動修正によって設計意図や責務分離が崩れていないか確認する。

---

## 禁止事項

以下は禁止する。

- 仕様にない機能を勝手に追加する
- `any` を安易に使う
- `console.log` を残す
- `panic` を通常のエラーハンドリングに使う
- access tokenや秘密情報をログに出す
- handlerにDB操作を書く
- repositoryに業務ロジックを書く
- ReactコンポーネントからDBに直接アクセスする
- API URLや秘密情報をコードにベタ書きする
- 認可条件なしでidだけを使って更新・削除する
- 大きすぎる関数やコンポーネントを放置する
- コメントと実装が矛盾した状態にする

---

## レビュー観点

レビュー時は以下を確認する。

### 共通

- 仕様書と矛盾していないか
- 単一責務の原則を守れているか
- コメントが目的や意図を説明しているか
- 命名が役割を表しているか
- 不要な複雑化をしていないか
- lint / format / test が通っているか

### セキュリティ

- `user_id` をリクエストから受け取っていないか
- 他ユーザーのデータを操作できないか
- 秘密情報をログやフロントエンドに出していないか
- バックエンド側で入力バリデーションしているか

### 保守性

- ファイルや関数が肥大化していないか
- 同じ処理が重複していないか
- DTOとdomain型が混ざりすぎていないか
- 変更理由が複数あるコードになっていないか

---

## AIエージェント向け作業ルール

CodexなどのAIエージェントが実装する場合は、以下を守る。

- 実装前に `docs/standards/agent-workflow.md` を読む
- 実装前に関連仕様書を読む
- 変更前に実装範囲を小さく区切る
- 一度に大きな変更をしない
- 仕様にない判断を勝手にしない
- 不明点がある場合は、実装前に確認する
- 実装後に変更ファイルと変更理由を要約する
- lint / format / test の実行結果を確認する
- 失敗したコマンドがある場合は、その理由を明記する

---

## この規約の更新方針

実装中に規約の不足が見つかった場合は、コード側で場当たり的に対応せず、この規約を更新する。

規約を変更した場合は、既存仕様書や実装との整合性も確認する。
